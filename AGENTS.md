# AGENTS.md

## 总体原则
- 仅在当前任务相关范围内工作，不要擅自扩大修改范围。
- 优先采用最小改动方案，不要无故重构。
- 不要为了“更优雅”而改动已经可用的代码。
- 不要修改与当前问题无关的文件、接口、命名和目录结构。
- 如存在多种实现方案，优先选择风险最小、侵入性最低的一种。

## 修改前要求
- 在修改代码前，先分析问题原因。
- 先指出可能涉及的文件、函数或模块。
- 先给出准备采取的修改思路，再实施修改。
- 如果信息不足，优先基于现有代码做最保守、最合理的推断，不要随意臆造不存在的机制。

## 修改时要求
- 只修改完成任务所必需的部分。
- 保持现有代码风格、命名风格、注释风格一致。
- 保持现有对外接口不变，除非任务明确要求修改接口。
- 不要随意重命名变量、函数、宏、文件。
- 不要无故移动文件、拆分文件、合并文件。
- 不要引入不必要的新依赖、新库、新抽象层。
- 不要一次性同时修改多个子系统，除非问题确实跨模块耦合。
- 修改逻辑时，尽量保留原有流程结构，避免造成阅读困难。

## 禁止事项
- 禁止大规模重构，除非我明确要求。
- 禁止顺手修复无关问题。
- 禁止只因为“代码不够漂亮”就重写。
- 禁止擅自删除已有日志、注释、保护代码、错误处理逻辑。
- 禁止在没有依据时声称“问题已彻底解决”。
- 禁止把猜测当作结论输出。

## 输出要求
- 每次修改后，说明修改了哪些文件。
- 说明每处修改的目的和原因。
- 说明该修改为什么能解决当前问题。
- 明确指出可能的风险点或潜在副作用。
- 给出我可以手动执行的验证步骤。
- 如果无法完全确认结果，要明确说明不确定性。

## 分析要求
- 发现问题时，优先先定位根因，而不是直接给补丁。
- 如果存在多个可能根因，请按可能性从高到低排序。
- 如果某个结论依赖假设，请明确写出假设条件。
- 对于关键修改，尽量解释“机制”，而不是只描述“现象”。

## 代码风格要求
- 保持实现直接、清晰、可读，不要炫技。
- 优先使用易维护的写法，不要过度简化成难读的表达。
- 注释应简洁、准确，聚焦作用和原因，不写空话。
- 对已有风格保持一致，不强行统一整个项目风格。
- 新增代码应尽量与周围代码风格一致。

## 任务执行顺序
在处理任务时，默认按以下顺序进行：

1. 理解需求
2. 分析可能原因
3. 确定影响范围
4. 提出最小修改方案
5. 实施修改
6. 总结修改内容
7. 给出验证方法
8. 提醒可能风险

## 当需求不明确时
- 默认选择范围最小、风险最低的实现。
- 明确写出你所采用的理解和假设。
- 不要自行扩展需求。
- 不要把“可选优化”混入“必要修改”。

## 当我要求你修 Bug 时
- 先定位最可能的根因。
- 不要一开始就大改。
- 优先尝试局部修复。
- 修复后给出复现、验证、回归检查建议。

## 当我要求你加功能时
- 先说明需要改哪些位置。
- 优先复用现有结构，不要平地起新框架。
- 不要因为新增功能而顺带重构旧模块，除非确有必要。
- 保证新增内容尽量不影响现有功能。
- 每次新增功能后，必须同步更新 `README.md`，说明新增功能、涉及模块、使用或验证方式。

## 当我要求你优化代码时
- 先说明当前问题是性能、可读性、冗余，还是结构问题。
- 只优化目标相关部分，不扩大到全局。
- 优化后说明收益和代价。
- 如果收益不明显，不要做高风险改动。

## 沟通风格要求
- 回答要具体，不要空泛。
- 结论要有依据，不要只给笼统判断。
- 不确定就明确说不确定。
- 不要使用“应该可以”“大概没问题”这类无依据表述，除非同时说明原因和前提。
